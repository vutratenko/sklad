# ADR-002: Границы модулей и ownership инвариантов

## Status

Accepted

## Context

Без явных границ монолит быстро превращается в «комбайн» с переплетённой логикой.

## Decision

Зафиксировать MVP-модули и правила зависимостей:

| Модуль | Ответственность | Инварианты |
|--------|-----------------|------------|
| `catalog` | SKU, категории, единицы, barcode | Уникальность barcode |
| `topology` | Склады, места хранения | Location принадлежит warehouse |
| `lots` | Партии, сроки годности | Lot привязан к SKU |
| `movements` | Операции по остаткам | Журнал движений, причина расхода |
| `stockview` | Read-модель остатков | Только чтение, без write-логики |
| `sync` | Транспорт offline-операций | Идемпотентность, без дублирования доменных правил |
| `auth` | OIDC, dev bypass | Fail-closed в production |

Модули общаются через application-service интерфейсы. Прямой доступ к таблицам другого модуля запрещён.

## Consequences

**Плюсы:** контролируемая связность, предсказуемая эволюция.

**Минусы:** требуется дисциплина при добавлении новых зависимостей.

## Rejected Alternatives

- Общий «анемичный» service layer без границ
- Прямой SQL-доступ между модулями
